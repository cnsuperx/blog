visual studio2022的版本对应是msvc-14.3(即v143工具集)

2.下载boost源码

下载地址：boost官网：https://www.boost.org/

3.生成构建工具

使用visual stido2022的Developer Command Prompt for VS 2022工具：

构建，生成 b2.exe ：

4.执行 b2 stage 编译

  • 参数说明：
    • toolset=msvc-14.3：指定 VS2022 编译器
    • address-model=64：生成 64 位库
    • –build-type=complete：编译所有变体（Debug/Release、静态/动态）
    • stage：仅生成库文件（.lib 和 .dll），不安装头文件

  1. 临时文件（编译中间产物）​

  2. 最终生成的库文件
    • Boost源码目录\stage\lib
  3. 头文件
    • Boost源码目录\include\boost-<版本>

6. 在 VS2022 中配置 Boost​

  1. ​​添加包含目录​
    • 右键项目 → 属性 → ​​C/C++ → 常规 → 附加包含目录​​ → 添加 Boost 根目录：D:\boost_1_87_0
  2. 添加库目录
    • 右键项目 → 属性 → ​​链接器 → 常规 → 附加库目录​​ → 添加：D:\boost_1_87_0\stage\lib

第二篇：流程图的制作与实战

篇章目标：全面讲解流程图在产品管理中的价值与用途，推荐主流绘图工具，深入剖析流程图绘制的思路和方法，并通过“电商退款＋多通道审批”场景，详细演示一份完整且具备一定复杂度的流程图实操案例。

一、为什么要画流程图？

  1. 厘清逻辑脉络
    任何产品功能背后都有一系列步骤、分支与并行关系，流程图能将这些复杂节点可视化，让团队迅速理解整体逻辑。

  2. 分工与协作
    产品、研发、测试、运营等多方协作时，流程图可以明确责任边界，避免“沟通游戏”与重复工作。

  3. 优化与迭代
    可视化流程更容易发现瓶颈、冗余与异常分支，为后续优化提供依据。

  4. 文档留存与培训
    在项目交接或新成员培训时，流程图比文字说明更直观，提高学习效率。

二、主流流程图工具推荐

工具 核心优势 适用场景
draw.io/diagrams.net 免费、支持云端或本地保存、开源、与版本控制系统（Git）无缝集成 中小团队或开源项目
Microsoft Visio 符号库丰富、与 Office 生态贴合、模板齐全 大型企业内部、需要导出多种格式
Lucidchart 在线协作流畅、支持实时评论与版本历史、与 Google Workspace／Slack 集成 跨地域跨团队实时协作
Miro / Whimsical 白板式自由绘制、支持脑图／线框／流程一体化；多人同步编辑 头脑风暴、快速原型、远程工作坊
BPMN 专业工具 完整遵循 BPMN 2.0 标准，能够进行流程模拟与审计 需要合规审计、流程治理的大规模企业或机构

选型建议：

  • 预算敏感、偏好开源：首选 draw.io；
  • Office 重度用户：Visio 更易上手；
  • 强协作需求：Lucidchart 或 Miro；
  • 合规与审计：BPMN 专业工具。

三、流程图绘制的核心步骤

1. 明确目标与边界

  • 起点与终点：定义流程从 “事件触发” 到 “结果交付” 的清晰边界。
  • 受众角色：谁需要阅读和使用这张图？技术团队、运营团队还是管理层？决定图的深度与粒度。

2. 梳理流程节点

  • 步骤列表：用文本或表格先列出所有活动（如“提交申请”、“审核判断”）和决策点（如“是否通过？”）。
  • 分支与并行：标注哪些步骤可以并行、哪些需要先后顺序；哪些决策会引入分支回流。

3. 选用标准符号

符号类型 图形示例 含义
起止 椭圆 / 圆角矩形 流程的开始与结束
操作 矩形 必须执行的业务操作或功能
决策 菱形 带分支输出的判断节点（是 / 否）
文档 / 数据 平行四边形或特殊图标 生成或使用的一类文档、表单或数据
并行 / 同步 双竖线 / 并行符号 表示并行流程
注释 云形框 对某个步骤的补充说明

要点：坚持使用统一符号并在图例中说明，保持团队阅读无障碍。

4. 绘制布局与连线

  • 流向一致：自上而下或自左向右，整个图保持一致方向。
  • 减少交叉：尽量避免线条相互穿插；必要时使用桥接（跳线）或拐角。
  • 留白合理：节点间留出足够间距，保证可读性和后续扩展。

5. 加色与标注

  • 泳道背景：为不同角色或部门泳道设置浅色背景，快速区分界限。
  • 高亮关键节点：对重要决策或异常回退分支，使用醒目颜色或加粗边框。
  • 分支文字：在连线上添加“是 / 否”、“通过 / 驳回”等标签。

6. 评审与优化

  • 版本控制：给流程图标注版本号、作者与更新时间；将文件纳入版本管理。
  • 多人评审：在画布上或会议中，邀请相关人士给出反馈并进行调整。
  • 定期复盘：将流程图作为持续改进的活文档，辅助团队迭代最佳实践。

四、实战案例：电商平台“退款＋多通道审批”流程

下面，我们以 draw.io 为工具，在“用户→客服→仓储→财务”四大角色泳道下，绘制一份较为完整的电商退款流程。

点击查看 Mermaid 渲染示例（仅示意）

实操要点解读：

  1. 泳道搭建

    • 使用 draw.io 左侧“Swimlane”组件，添加四条泳道，分别命名。
    • 为每条泳道设置浅灰色／浅蓝色背景，突出分工边界。

  2. 节点绘制与连线

    • 用户泳道：申请退款 → 上传凭证。
    • 客服泳道：接收请求 → 判断凭证完整性 → 初步审核 → 根据订单状态分支。
    • 仓储泳道：仅在“已发货”分支下绘制，完成寄回验货后分为“合格 / 不合格”循环。
    • 财务泳道：直接从“直接退款”与“合格入库”两个节点汇集，生成退款单并支付。

  3. 决策与回流

    • 决策节点均使用菱形，连线标注“是/否”或“已发货/未发货”以消除歧义。
    • 回流：凭证不全时，从客服泳道“通知补充”回流到用户“申请退款”节点，形成闭环。

  4. 异常与并行

    • 并行执行的概念在此流程中较少，但可在后续迭代中将“财务核对”与“客服确认”并行。
    • 异常分支（如“拒收并退回”）可加背景色或红色边框，突出提醒。

  5. 注释与版本

    • 在画布一角添加文本框，标注 “版本：v1.0 2025-06-20 by PM 王伟”，方便后续追溯。
    • 如需扩展，可以新增“法务审核”或“风控评估”泳道，保证可扩展性。

五、总结与实践建议

  • 先画后说：在任何需求讨论会前，先用草图（手绘或白板）快速勾勒流程，再转入工具精绘。
  • 业务+技术并重：流程图不仅要体现业务逻辑，也要兼顾系统实现（如并发、自动化触发）。
  • 可执行的文档：将流程图与相关 SOP、操作手册结合，确保“图说流程”能真正落地执行。
  • 持续迭代：将流程图当成活文档，定期收集实际执行中的痛点数据，不断优化分支与并行方案。

认识互联网与产品经理

Posted on 2025-06-20 In 产品经理教程01

第一篇：认识互联网行业与产品经理

本章目标：带你快速了解互联网行业全貌、企业组织架构与各岗位职责，厘清产品经理（PM）的角色定位与核心工作内容，并梳理PM必备的技术与知识体系，为后续学习打下坚实基础。

一、互联网行业概述

  1. 行业特点

    • 高速迭代：从需求、竞品到技术，实现周期通常以“周”为单位，强调持续交付。
    • 用户驱动：一切决策以用户价值为中心；通过数据、调研及时校准产品方向。
    • 多元生态：包含电商、社交、内容、金融科技、企业服务、硬件+软件等多种形态；跨界融合趋势明显。
    • 技术赋能：云计算、大数据、AI、区块链等新技术不断渗透，产品创新不断涌现。

  2. 行业发展阶段

    • 探索期（2000–2010）：PC 互联网兴起，门户与搜索业务为主；移动端萌芽。
    • 爆发期（2010–2015）：智能手机普及，移动互联网全面铺开；社交、电商、O2O 快速扩张。
    • 成熟期（2015–至今）：深度服务化、平台化、生态化，流量红利减弱，竞争加剧，向下沉市场和海外拓展。

二、互联网公司组织架构与岗位分工

典型架构图（参照大中型互联网公司）

部门 主要职责
产品中心 规划产品战略、定义需求、制定迭代计划、跨部门协调推进
技术中心 实现产品功能、保证系统稳定与性能、安全建设
运营中心 用户拉新留存、内容/活动运营、数据监控与优化
市场与品牌 品牌传播、市场推广、用户洞察与竞品分析
销售与商务 商业合作、广告/流量变现、渠道拓展
支撑职能 人才招聘与培养、财务预算与审核、合同与合规、后勤保障

三、各关键岗位工作内容

  1. 产品经理（PM）

    • 用户需求洞察：调研用户痛点、挖掘场景，形成产品定位与价值主张。
    • 产品规划与设计：撰写 PRD（产品需求文档）、流程图、线框图；制订里程碑与迭代计划。
    • 跨部门沟通：与研发、测试、运营、市场、设计等协同，保障需求落地。
    • 进度管理与迭代优化：监控开发进度、上线效果，收集反馈、持续迭代。
    • 数据分析：定义关键指标（KPI/OKR），通过埋点与数据看板跟踪用户行为与产品健康度。

  2. 交互/视觉设计（UX/UI）

    • 交互设计：用线框图（Wireframe）、原型（Prototype）呈现用户流程与交互逻辑。
    • 视觉设计：品牌、色彩、排版、动效设计，确保界面美观易用。

  3. 研发工程师（前/后端、移动）

    • 前端/移动：根据设计稿实现界面与交互逻辑；保证兼容性与性能优化。
    • 后端：搭建业务服务、数据库设计、安全与权限控制；支撑高并发与大数据场景。

  4. 测试/QA

    • 功能测试：验证需求覆盖度与功能正确性；
    • 性能安全测试：压测、渗透，保障上线质量；
    • 自动化测试：构建自动回归测试、CI/CD 集成。

  5. 运营/市场

    • 内容/活动运营：策划主题活动、内容生产与审核；
    • 用户运营：拉新、促活、留存、转化；
    • 市场推广：广告投放、渠道合作、品牌传播；
    • 数据/BI：日常报表、A/B 测试、深度分析。

四、产品经理核心工作流程

  1. 需求收集与分析

    • 用户访谈、竞品调研、行业趋势研究
    • 整理与优先级评估（价值x成本矩阵、RICE 模型等）

  2. 方案设计与评审

    • PRD/用户故事、流程图、原型图
    • 召开需求评审会，收集各方意见并打磨

  3. 开发与测试推进

    • 制定迭代计划（Sprint/迭代周期）
    • 日常站会、燃尽图监控进度
    • 测试用例设计、BUG 跟踪与验证

  4. 上线与运营

    • 上线前灰度、回滚方案、上线发布
    • 监控数据：DAU/MAU、转化率、留存率、关键漏斗、业务指标
    • 运营活动配合、用户反馈渠道管理

  5. 复盘与迭代

    • 数据复盘、用户调研，形成迭代需求
    • 优化路径：功能改进、性能优化、体验升级

五、产品经理需熟悉的技术与知识体系

  1. 基础技术概念

    • Web 原理：HTTP/HTTPS、TCP/IP 基础、前后端分离模式
    • 移动端：iOS/Android 基础差异、常见适配方案
    • 数据库：关系型（MySQL/PostgreSQL）与非关系型（Redis/MongoDB）
    • 缓存与消息队列：Redis、Kafka/RabbitMQ 用例及选型

  2. 工具与方法论

    • 原型与设计工具：Axure、Sketch、Figma、PS
    • 项目管理工具：Jira、Trello、禅道、Tower、飞书协同
    • 数据分析工具：Excel、SQL、Looker/Data Studio、Python（Pandas 简析）
    • 敏捷与精益：Scrum、Kanban、OKR、KPI、用户故事地图、MVP 思维

  3. 行业与业务知识

    • 产品思维：需求 vs. 场景 vs. 痛点；价值主张与商业模式
    • 竞品与市场分析：SWOT、五力模型、蓝海战略
    • 增长黑客：AARRR 模型：获取（Acquisition）、激活（Activation）、留存（Retention）、推荐（Referral）、收入（Revenue）

  4. 软技能

    • 跨部门沟通：如何撰写有效邮件、会议主持、谈判与协调
    • 需求优先级：平衡业务诉求与技术成本
    • 数据思维：用数据说话，避免主观臆断
    • 用户同理心：深度理解用户，才能设计出打动用户的产品

Google Play账号被封，重新上架建议

Posted on 2025-06-16 Edited on 2025-06-17 In Android

Google Play账号被封，重新上架建议

  1. 完全换用新的法人实体与帐号
    新公司注册：必须是真正独立的法人，注册号、法定代表人、营业地址、银行账户等与原账号毫无交集。

新 Google 帐号：用与原来完全不相关的邮箱（Gmail），从未与原账号有任何交互。

新支付资料：新的信用卡或银行账户信息，绝不使用原账号的任何付款方式。

  1. 彻底隔离签名与包名
    全新 keystore：重新生成 .keystore/.jks，切勿沿用旧的任何参数。

全新包名：使用与旧包名风格完全不同的命名空间，避免任何前缀或变体相似。

  1. 彻底重构代码与资源
    代码级别混淆/重写：不仅仅是 ProGuard 混淆，更要重构关键逻辑（如启动页、UI 样式、业务流程），删除或替换所有可能被指纹识别的类名、布局 ID、字符串资源。

资源大规模替换：更换图标、启动图、UI 视觉元素，甚至调整配色和交互流程。
在商店截图、首图、图标等“可见”资源上做微调，可以让审核和用户更直观地区分新旧版本；
对游戏内一些关键 UI、启动页、logo 这类极易记住的元素做轻量化改动，也能在万一被人工抽检时降低被“关联”疑虑的风险。

  1. 全新后端与网络环境
    新域名／IP／服务器：游戏的后端服务（授权、数据上报、支付回调）使用全新的域名和服务器，且部署在与旧项目完全不同的网络环境或机房。

新证书与 TLS 配置：后端的 SSL 证书、API Key、OAuth 客户端都要全新生成。

  1. 客服与用户联系方式
    全新联系方式：申请新的客服电话、邮箱、社交媒体账号，且绝不复用原有渠道。

隐私政策与协议：重新起草并托管在全新域名下。

  1. 开发与测试环境隔离
    独立开发网络：在不同的办公网络或 VPN 下进行开发、构建、上传，避免和原项目共用任何 IP。

全新测试设备：不要在之前用来测试旧版本的设备上安装新版，避免设备指纹关联。

  1. 发布流程与渠道
    仅走 Google Play：务必通过 Google Play Console 上传测试，再测试账号下载安装，切勿绕开商店直接安装。

内部/封闭测试轨道：先在内部测试轨道跑一轮，通过后再推送到生产。

⚠️ 最后
即便做到上述所有隔离，Google Play 的风控依旧可能通过更深层的信号（如用户行为模式、崩溃日志特征、包体哈希、API 调用序列）将您和被封应用关联起来，一经发现，“新”帐号与“新”应用同样会被迅速终止。
